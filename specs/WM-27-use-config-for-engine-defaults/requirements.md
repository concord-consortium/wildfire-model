# Hazbot: Source Engine Defaults from the Simulation Config

**Jira**: https://concord-consortium.atlassian.net/browse/WM-27
**Repo**: https://github.com/concord-consortium/wildfire-model
**Implementation Spec**: [implementation.md](implementation.md)
**Status**: **In Development**

## Overview

<!-- Rewritten during Finalization. -->
The Hazbot analysis engine decides whether a student changed a simulation
variable by comparing logged `SimulationStarted` events against a "defaults"
object. Today that object is hand-extracted, per rule-set, from a spreadsheet's
Details column — and when the extraction comes up empty the engine refuses to
load the rule-set. This work makes the engine derive its change-detection
defaults directly from the resolved simulation config (preset + URL params),
which is the actual initial state the simulation loads, and removes the
per-rule-set defaults object and the missing-defaults failure mode.

## Project Owner Overview

<!-- Rewritten during Finalization. -->
The Hazbot gives students feedback based on what they changed in the wildfire
model. To know what "changed" means, it needs to know the model's starting
state. Currently that starting state is transcribed by hand from a planning
spreadsheet into each rule-set, which is brittle: when the transcription is
incomplete, the Hazbot silently turns off for that activity. This change has
the Hazbot read the starting state from the same configuration the simulation
itself uses, so every activity's Hazbot works without manual data entry, and a
class of "Hazbot won't load" failures disappears. It also unblocks WM-18, which
needs the engine to load freshly-extracted rule-sets.

## Background

### How change-detection works today

The wildfire bridge (`src/hazbot/wildfire/`) defines factor variables —
predicates over the recorded event log. Six of them detect *change* against a
default:

- `setTerrainType`, `setVegetation`, `setDroughtLevel` — true if any zone in any
  `SimulationStarted` reading differs from the default for that zone/field.
- `setWind` — true if any reading's wind speed/direction differs from the default.
- `setAnyZoneVar` (= terrain OR vegetation OR drought) and `setAnyVar` (= those
  plus wind) — composite predicates over the same comparisons.

Each of these declares `requiredDefaults` (e.g. `["zones[*].droughtLevel"]`) and
its `compute(readings, defaults)` reads the default value to compare against.

The default value comes from `RuleSet.defaults` — a `WildfireDefaults` object
(`{ zones?: ZoneDefaults[]; wind?: { speed; direction } }`) baked into each
generated rule-set module under `src/hazbot/rule-sets/` (`23.ts`, `24.ts`, …).

### Where the defaults object comes from today

The rule-set modules are auto-generated by `scripts/extract-hazbot-sheets.js` /
`scripts/extract-impl.js` from a source spreadsheet. The extractor's
`mergeDefaults` / `parsePerZoneDefault` / `parseWindDefaults` functions scan the
free-text **Details** column of each factor-variable row with regexes (looking
for phrases like `Default values = "Plains" (zone 1), "Plains" (zone 2)`) and
assemble a `defaults` object. When the Details text says `TBD (activity
revision)` — or simply doesn't match the regex — no default is emitted and the
rule-set ships with `defaults: {}`.

### The failure mode being removed

At engine load time, `Engine.collectFromImpl()` validates every referenced
impl's `requiredDefaults` paths against `RuleSet.defaults` via
`validateDefaultsPath()`. A path that doesn't resolve produces a
`load-failure / missing-defaults` `EngineError`. `missing-defaults` is a
load-blocking error, so `engine.isActive` is false and the engine refuses to
process events — the Hazbot is effectively off for that activity.

Rule-sets **32, 33, 34, 35** currently ship with `defaults: {}` *and* reference
`set*` factor variables, so they fail to load today. (Rule-set **25** also has
`defaults: {}` but references no defaults-bearing factor variable, so it loads
fine. Rule-sets **23** and **24** ship with fully-populated `defaults`.)

**Impact scope.** The Hazbot is in early development; no students are using it
yet. It is presently surfaced only through the developer/researcher sidebar
(`hazbotSidebar`), and `hazbotRules` becomes a production, student-facing flag
only once the Hazbot UI lands. So today this failure means rule-sets 32–35
cannot be exercised in the dev sidebar or research tooling — there is no
live student-facing breakage — and removing it also ensures those rule-sets
will work when the student-facing UI ships.

**A trade this spec mitigates.** Removing `missing-defaults` swaps a *loud*
failure (engine inactive, visibly off) for a *silent* one: if an activity's LARA
URL selects the wrong preset, the engine still loads and derives defaults from
that wrong preset. Requirement 13 (*Observability*) is scope this spec adds to
offset that trade — it is **not** part of the WM-27 ticket text. It is kept in
WM-27 rather than split to a follow-up ticket because it directly mitigates a
regression this same story introduces, and the work (one log-payload field plus
a small dev-sidebar display) is too small to justify separate-ticket overhead.

### Why config is the right source

The values the `set*` factor variables compare against are nothing more than the
simulation's *initial configuration*. The `SimulationModel` resolves its config
as `Object.assign(getDefaultConfig(), presetConfig, getUrlConfig())`
([src/models/simulation.ts:162](../../src/models/simulation.ts#L162)) — the base
defaults from [src/config.ts](../../src/config.ts), overlaid with the selected
preset from [src/presets.ts](../../src/presets.ts), overlaid with URL-param
overrides. That resolved config already carries the initial per-zone
`terrainType` / `vegetation` / `droughtLevel` and the initial `windSpeed` /
`windDirection`. The engine should derive `WildfireDefaults` from that same
config instead of from a hand-transcribed spreadsheet column.

### Spreadsheet verification (config defaults vs. the Details column)

The current source spreadsheet
(`Wildfire-Hazbot-Feedback-Tables-2026-05-21.xlsx`) documents each activity's
initial condition on a dedicated **SIMINIT** sheet, and the per-rule Details
columns now mostly *point to* that sheet ("For the default values, read the
SIMINIT sheet") rather than restating values inline. Cross-checking SIMINIT
against the presets in `src/presets.ts` (drought levels: `MildDrought = 1`):

| Rule-set | SIMINIT initial condition | Matching preset | Match? |
|---|---|---|---|
| 23 | 2 zones — Plains/Shrub/Mild ×2; wind 0/0 | `plainsTwoZone` (Plains/Shrub/1 ×2) | ✓ |
| 24 | 2 zones — Plains/Shrub/Mild ×2; wind 0/0 | `plainsTwoZone` | ✓ |
| 25 | 2 zones — Mountains/Shrub/Mild ×2, terrain fixed | `mountainTwoZoneFixedTerrain` (Mountains/Shrub/1 ×2) | ✓ |
| 32 | 3 zones — Plains/Grass/Mild ×3, terrain fixed | `threeGreenZonePlains` (Plains/Grass/1 ×3) | ✓ |
| 33 | 2 zones — Mountains/Shrub/Mild ×2 | `mountainTwoZone` (Mountains/Shrub/1 ×2) | ✓ |
| 34 | 3 zones — Mtn/Shrub/Mild, Foothills/Shrub/Mild, Plains/Shrub/Mild, terrain fixed | `shrubThreeZone` | ✓ |
| 35 | 2 zones — Mountains/Shrub/Mild ×2 | `mountainTwoZone` | ✓ |

The currently-extracted `defaults` in `23.ts` and `24.ts` also match SIMINIT.
**Conclusion:** for every current rule-set, a preset exists in `src/presets.ts`
whose zone/wind values match the spreadsheet-documented initial condition, so
config-derived defaults are equivalent to the (correct) hand-extracted ones.
*Caveat:* this verifies that matching presets exist; it does **not** verify that
each activity's LARA URL actually selects the matching preset — see RESOLVED OQ4.
Relatedly, if a LARA URL names a `preset` that matches no key in
`src/presets.ts` (typo, renamed preset), `getResolvedConfig()` silently falls
back to the base config and derives defaults from it — the engine no longer
fails closed on defaults, so this is not flagged as an error (see the
Self-Review *Education Researcher* item on making the requested preset
auditable).

### Reading format note

The `SimulationStarted` payload reports zones with **string labels**
(`vegetationLabels[z.vegetation]` → `"Shrub"`, etc.) and wind as
`{ speed, direction, scaleFactor }`
([src/components/bottom-bar.tsx:237-246](../../src/components/bottom-bar.tsx#L237-L246)).
The config, however, stores zones with **enum values** (`TerrainType.Plains = 0`,
`Vegetation.Shrub = 1`, `DroughtLevel.MildDrought = 1`). Config-derived defaults
must therefore be converted to the same label strings the readings use, so the
factor-variable comparisons are like-for-like.

## Requirements

### Config-derived defaults

1. A shared `getResolvedConfig()` helper is added to `src/config.ts`. It returns
   the full resolved config — the merge `Object.assign(getDefaultConfig(),
   preset, getUrlConfig())`, with the preset resolved from the URL as
   `presets[getUrlConfig().preset || getDefaultConfig().preset]`. It accepts an
   optional explicit preset partial; when one is supplied the helper uses it in
   place of the URL-resolved preset, so a caller can still resolve a config
   around an injected preset (the `SimulationModel` constructions in
   `simulation.test.ts` rely on this). The explicit preset partial substitutes
   **only** the preset slot of the merge: the result is still
   `Object.assign(getDefaultConfig(), explicitPreset, getUrlConfig())`, so
   `getUrlConfig()` overrides continue to apply on top. The helper has exactly
   one runtime-resolution behaviour — the explicit-preset parameter changes
   *which* preset is merged, never *whether* the base and URL layers are
   applied. There is no separate "compose two arbitrary partials" mode.
   `src/models/simulation.ts` and the
   wildfire bridge both route through this helper for the full merge.
   `src/models/stores.ts` no longer resolves a preset itself: its duplicated
   `getUrlConfig().preset || getDefaultConfig().preset` expression is deleted and
   it constructs `SimulationModel` with no argument, because the helper resolves
   the preset from the URL. The `SimulationModel` constructor / `load()`
   `presetConfig` parameter becomes optional, retained only so tests can inject a
   config. The merge and the preset-name resolution thus live in one place
   instead of being re-coded at each call site. The merge is intentionally a
   **shallow** `Object.assign`: each top-level config key — including the
   `zones` tuple — is taken wholesale from the highest-priority source that
   defines it (URL over preset over base defaults), with no per-zone or other
   nested deep merge. A preset that defines `zones` therefore replaces the base
   `zones` entirely. This preserves the existing `simulation.ts` merge behaviour
   exactly; centralizing it in `getResolvedConfig()` means the shallow semantics
   are defined in one place rather than re-cited at each call site. Throughout
   this spec, "**resolved config**" (and "full resolved config") refers to
   exactly this merge result — the value `getResolvedConfig()` returns. It never
   denotes a bare preset partial or a URL-config fragment.
2. The wildfire bridge derives `WildfireDefaults` (per-zone `terrainType` /
   `vegetation` / `droughtLevel`, plus wind `speed` / `direction`) from the
   resolved config returned by `getResolvedConfig()`.
3. Derived zone defaults are expressed as the same string labels the
   `SimulationStarted` payload uses (`terrainLabels` / `vegetationLabels` /
   `droughtLabels`), with one default derived per populated entry of the
   resolved `config.zones` tuple, in tuple order (independent of `zonesCount`),
   so factor-variable comparisons are like-for-like. Deriving a default for
   every config zone is safe even when a run uses fewer zones: `anyZoneDiffers`
   is driven by the reading's zones and ignores surplus defaults. This is a
   structural invariant, not an assumption about upstream correctness: a
   `SimulationStarted` reading's `zones` array is produced by the same
   `SimulationModel` instance whose resolved config supplied the derived
   defaults, so a reading cannot enumerate more zones than `config.zones`
   defines. A reading carrying more zones than the derived defaults would
   indicate a corrupted event log, which is outside this engine's contract; the
   derivation defines no fallback for it.
4. Derived wind defaults are `{ speed: config.windSpeed, direction:
   config.windDirection }`.
5. `setTerrainType`, `setVegetation`, `setDroughtLevel`, `setWind`,
   `setAnyZoneVar`, and `setAnyVar` compare logged `SimulationStarted` events
   against the config-derived defaults, which reach `compute()` via a new
   engine-level `EngineOpts.defaults` input the bridge supplies at construction.

### Substrate removal (full)

6. `RuleSet.defaults` and `FactorVariableImpl.requiredDefaults` /
   `SimPropImpl.requiredDefaults` are removed from the engine substrate. The
   `DeepPartial` helper, retained only for `RuleSet.defaults`, is removed if it
   has no other use.
7. The `missing-defaults` failure mode and all its supporting machinery are
   removed: the `missing-defaults` reason in the `load-failure` `EngineError`
   variant — and the code that produces it — while the `load-failure` variant
   itself stays for its `missing-rule-set` / `missing-impl` reasons;
   `Engine.collectFromImpl()`, `Engine.implsWithIncompleteDefaults`,
   `validate-defaults.ts` (`validateDefaultsPath`), and the incomplete-defaults
   guard branch in `safely-evaluate-impl.ts`. The engine no longer rejects or
   degrades any rule-set for absent or incomplete defaults.
8. The engine loads every existing rule-set — 23, 24, 25, 32, 33, 34, 35 — with
   no load-blocking error attributable to defaults.
9. Rule-sets 23 and 24 — whose presets the Background's *Spreadsheet
   verification* table confirms match their documented initial conditions —
   continue to classify behaviour unchanged. Operationally: the existing
   per-rule-set classification suites `src/hazbot/rule-sets/23.test.ts` and
   `24.test.ts` pass unchanged once `RuleSet.defaults` is replaced by
   config-derived defaults supplied via `EngineOpts.defaults` (the suites'
   fixtures already encode the Plains/Shrub/Mild ×2 baseline; their passing
   confirms the config-derived defaults reproduce it). No test is added that
   pins config-derived defaults to the now-deleted hand-extracted `defaults`
   objects — see the QA Engineer Self-Review item.

### Extractor & generated modules

10. The sheet-based defaults-extraction logic is retired from
    `scripts/extract-impl.js` (`mergeDefaults`, `parsePerZoneDefault`,
    `parseWindDefaults`, the `FACTOR_VAR_TO_FIELD` map, the `defaults`/`finalDefaults`
    assembly in `parseTab`, `emitDefaults`, and the `TBD (activity revision)`
    handling). The generator no longer emits a `defaults` field.
11. The `defaults` field is removed from the 7 committed generated rule-set
    modules (`23.ts`–`35.ts`) by a direct surgical edit — **not** by
    regeneration — so no formatter, import-ordering, line-ending, or whitespace
    drift is introduced. Every line other than the removed `defaults` field
    stays byte-identical to what is currently committed; the diff for each
    module is exactly that field's deletion and nothing else, which is what
    makes the change reviewable as such. (No content refresh from the newer
    spreadsheet — that belongs to WM-18; see RESOLVED OQ2.)
12. A note is added to `docs/hazbot-update-workflow.md` recording that, after
    WM-27, the committed `src/hazbot/rule-sets/*.ts` modules are intentionally
    *not* a clean regenerate: regenerating them before WM-18 lands will
    reintroduce unrelated `details`/wording drift from the newer spreadsheet,
    because the surgical strip (RESOLVED OQ2) removed only the `defaults` field
    and left all other content at its older-sheet revision. WM-18 owns
    reconciling the modules with a clean regenerate.

### Observability

13. The bridge surfaces the preset name the activity URL **requested** and
    whether that name was **recognized** (matched a key in `src/presets.ts`),
    so a mis-binding is visible rather than silent. Terminology used throughout
    this requirement: the **requested preset** is the verbatim URL `preset`
    value; **recognized** means it matched a known preset; an unrecognized name
    causes a silent fallback to the base config. The requirement deliberately
    avoids "resolved preset" as a noun — for an unrecognized name there is no
    resolved preset, the baseline is simply the base config.
    - **Log payload.** When the URL provides a `preset` value, that value is
      added verbatim to the `AnalysisEngineActivated` log payload — together
      with a boolean indicating whether that value is **recognized** — i.e.
      matches a known preset in `src/presets.ts` (an unrecognized name silently
      fell back to the base config). This extends `buildAnalysisEngineActivatedPayload()`, which today
      carries only `engineVersion` / `appRulesVersion` / `ruleSetId`. Recording
      the raw value rather than a "resolved preset name" ensures an unrecognized
      preset — the mis-binding worth catching — is captured rather than dropped.
      This payload is emitted via `src/log.ts`'s `externalLog` path, so it
      reaches LARA's logging backend and the log-monitor like every other event
      — the unrecognized-preset case is therefore machine-detectable in logs,
      not only visible to a developer inspecting the sidebar. No separate
      console warning, telemetry counter, or `EngineError` is added: a
      load-blocking `EngineError` for an unrecognized preset was already
      rejected (the engine no longer fails closed on defaults), and the logged
      recognized-preset boolean is the structured signal a log consumer queries
      on. *Limitation:* the `AnalysisEngineActivated` event is emitted only when
      the engine activates (a recognized `hazbotRules` rule-set loads); an
      activity whose engine never activates — no `hazbotRules`, or a typo'd
      `hazbotRules` — logs no such event, so a mis-bound `preset` there is caught
      by the dev sidebar (whose diagnostics are ungated on the rule-set) but not
      by logs.
    - **Dev sidebar.** When the URL provides a `preset` value, the dev sidebar
      displays it, styled with the sidebar's existing match/no-match
      convention: green when the value is a **recognized** preset in
      `src/presets.ts`, red with strikethrough (the `hazbot-sidebar-leaf-false`
      treatment) when it does not — an unrecognized name that fell back to the
      base config (see the *Spreadsheet verification* caveat). That convention
      is double-encoded (colour + underline/strikethrough) for *colour-blind*
      accessibility, but `text-decoration` is not announced by screen readers
      and the leaf classes carry no text equivalent. So the no-match state must
      additionally carry a **text** cue: the displayed name appends a short
      literal label (e.g. ` (unrecognized preset)`) so the base-config fallback
      is conveyed without relying on styling — announced by screen readers and
      visible regardless of colour or decoration. The display also carries a
      short identifying label (e.g. "Requested preset:") preceding the value,
      following the sidebar's existing label convention, so the field — and,
      with the text cue, its match/no-match state — is intelligible without
      relying on position or styling.
    - When the URL provides no `preset` param, the sidebar shows nothing — there
      is no author-supplied value to validate.

    Requirement 13 surfaces the *requested* preset name, not the full derived
    `WildfireDefaults`. The engine's defaults derive from the whole resolved
    config, so an activity configured via URL params alone (no `preset`), or a
    preset plus heavy URL overrides, can still derive a wrong baseline that this
    observability will not catch. Surfacing the derived `WildfireDefaults`
    directly is deliberately deferred: the Hazbot is dev-only with no students
    yet (see the Background *Impact scope* note), so the research-data-validity
    urgency is low today. The preset name is accepted as a sufficient, cheap
    first signal for now, and surfacing the full derived defaults is a natural
    candidate to revisit when the Hazbot becomes student-facing.

### Quality gates

14. Affected unit tests are updated; `npm test`, `npm run lint`, and
    `npm run build` all pass.

    `compute`/`evaluate`. The `defaults` source changes from `ruleSet.defaults`
    to an engine-level value.
  - [scripts/extract-impl.js](../../scripts/extract-impl.js) — extraction +
    emission; `extract-impl.test.js` covers it.
  - [src/config.ts](../../src/config.ts) — `getDefaultConfig()` / `getUrlConfig()`.
    The merge `Object.assign(getDefaultConfig(), preset, getUrlConfig())`
    currently lives in `simulation.ts`; `stores.ts` separately duplicates only
    the preset-name resolution `getUrlConfig().preset || getDefaultConfig().preset`
    it feeds to `SimulationModel`.
- **Config resolution.** `getAnalysisEngine()` would need preset + base config,
  not just `getUrlConfig()`. The merge is `Object.assign(getDefaultConfig(),
  presets[getUrlConfig().preset || getDefaultConfig().preset], getUrlConfig())`.
- **Zone count.** The resolved config's `zones` is a fixed-arity tuple
  (`[Zone, Zone, Zone?]`) and `zonesCount` (`2 | 3 | undefined`) determines how
  many are live. The derivation emits one default per populated entry of
  `config.zones`, regardless of `zonesCount` — that count is known at
  config-resolution time and is never shorter than any reading. `anyZoneDiffers`
  iterates the *reading's* zones and skips when the default for that index is
  `undefined`, so surplus derived zones are harmless; a too-short defaults list
  is the only failure mode, and deriving from the full `config.zones` tuple
  avoids it.
- **Wind.** `setWind` compares `speed` and `direction` only; `windScaleFactor`
  is not part of the comparison. All current rule-sets (23–35) have wind 0/0.
- **Substrate genericity.** The `hazbot/engine` layer is documented as a generic
  substrate; the `hazbot/wildfire` bridge is its only consumer. Removing
  `RuleSet.defaults` / `requiredDefaults` / `validate-defaults.ts` changes the
  substrate surface — see OQ1. This is deliberately a *relocation*, not a
  narrowing: the engine keeps a generic default-injection channel in
  `EngineOpts.defaults`, which any future consumer (wildfire or otherwise) can
  supply at construction. What is removed is specifically (a) `RuleSet`-baked
  *static* defaults and (b) load-time *path validation* of them. The post-WM-27
  substrate philosophy: defaults are an engine-construction input, and ensuring
  they are complete and well-formed is the consumer's responsibility, not the
  substrate's. A future consumer that wants static or validated defaults would
  re-add that as a new, separately-justified feature rather than inherit dead
  machinery kept "just in case."
- **Derivation does no validation, by design.** The config → `WildfireDefaults`
  derivation reads `terrainType` / `vegetation` / `droughtLevel` and
  `windSpeed` / `windDirection` straight from the resolved config, converting
  enum values via the *total* `terrainLabels` / `vegetationLabels` /
  `droughtLabels` maps. It adds no defensive checking. The config's zone shape
  (`ZoneOptions` in `src/models/zone.ts`) types these fields loosely —
  `terrainType?` / `vegetation?` are optional and `droughtLevel?` is `number`,
  not `DroughtLevel` — so the derivation casts (`as TerrainType` /
  `as Vegetation` / `as DroughtLevel`) to index the label maps. The casts are
  sound, not defensive: `getDefaultConfig()` and every preset in
  `src/presets.ts` populate all three zone fields and are TypeScript-checked, so
  a resolved config's zones always carry valid enum values; the casts merely
  assert a guarantee the `ZoneOptions` type is too loose to express, and add no
  runtime check. The only way to inject a malformed value is a
  hand-crafted URL param (`?zones=…`) lying about the JSON shape
  `getUrlConfig()` casts unchecked — and that same malformed config flows into
  `SimulationModel.config` and visibly breaks terrain rendering, independent of
  the Hazbot. A malformed config is therefore a pre-existing, loud failure of
  the simulation itself; the derivation deliberately adds no redundant
  Hazbot-side assertion, consistent with the post-WM-27 substrate philosophy
  that completeness and validity are the consumer's contract, not a
  re-introduced validation layer.
- **Sim-props** (`sim-props.ts`) carry a `WildfireDefaults` type parameter but
  none of them read defaults, so they are behaviourally unaffected.
- **Affected tests** (non-exhaustive): `factor-variables.test.ts`,
  `engine.test.ts`, `validate-defaults.test.ts`, `safely-evaluate-impl.test.ts`,
  `error-rendering.test.ts`, `sidebar.test.tsx`, `consume.test.ts`,
  `replay-fixture.test.ts`, `rule-sets/*.test.ts`, `test-helpers.ts`,
  `extract-impl.test.js`. New coverage is also needed for the net-new config →
  `WildfireDefaults` derivation this story introduces — exercising a 2-zone
  config, a 3-zone config, the enum→label conversion, and the per-zone count

  The shared `set*`
  compute logic itself is not per-rule-set, so it needs no per-rule-set test

## Open Questions

### RESOLVED: How far does the substrate cleanup go?
**Context**: The ticket names `engine.ts` explicitly and says "the
missing-defaults failure mode [is] removed." But the `hazbot/engine` layer is a
generic substrate, and `requiredDefaults` / `validateDefaultsPath` /
`implsWithIncompleteDefaults` / the `defaults`-undefined guard in
`safely-evaluate-impl.ts` are all generic machinery. Deciding how much to delete
changes the diff size and what "done" means.
**Options considered**:
- A) **Full removal.** Delete `RuleSet.defaults`, `FactorVariableImpl.requiredDefaults`,
  `validate-defaults.ts`, the `missing-defaults` reason in the `load-failure`
  `EngineError` variant, `collectFromImpl`, `implsWithIncompleteDefaults`, and the
  render-path incomplete-defaults guard. Defaults reach `compute()` via a new engine-level
  channel (e.g. `EngineOpts.defaults`). Smallest long-term surface; largest diff.
- B) **Keep the substrate generic, drop only the failure mode.** Defaults become
  an `EngineOpts`-level input instead of `RuleSet.defaults`; keep
  `requiredDefaults` as optional metadata but remove the load-blocking
  `missing-defaults` error. Medium diff; substrate stays reusable.
- C) **Minimal.** Keep `RuleSet.defaults`; have the wildfire bridge inject
  config-derived defaults into each `RuleSet` object before engine construction;
  guarantee completeness so `missing-defaults` never fires (but leave the code
  path in place). Smallest diff; leaves dead-ish code.

**Decision**: **A — Full removal.** Once defaults are always config-derived they
are always complete, so `requiredDefaults`, `validateDefaultsPath`, the
`missing-defaults` error, `collectFromImpl`, `implsWithIncompleteDefaults`, and
the render-path incomplete-defaults guard have no remaining job — keeping them
would be dead code and a maintenance trap. Defaults reach `compute()` via a new
engine-level channel: `EngineOpts.defaults`, a single engine-level value the
wildfire bridge supplies at construction. `compute(readings, defaults)` /
`evaluate(reading, defaults)` signatures stay intact.

### RESOLVED: How should the `defaults` field be removed from the generated rule-set modules?
**Context**: `src/hazbot/rule-sets/*.ts` are auto-generated and currently
committed from an *older* spreadsheet revision — their `details` text and some
category wording differ from the 2026-05-21 spreadsheet (e.g. `23.ts` says
`Default values (trust but verify) = …` where the new sheet says `see the
SIMINIT sheet`). Removing the `defaults` field means regenerating, and a full
regenerate would also pull in that unrelated content drift.
**Options considered**:
- A) **Surgical strip.** Update the generator so it no longer emits `defaults`,
  then remove just the `defaults` field from the 7 committed modules, leaving all
  other generated content byte-identical to what is committed.
- B) **Full regenerate** from `Wildfire-Hazbot-Feedback-Tables-2026-05-21.xlsx` —
  removes `defaults` and refreshes all other extracted content in one step
  (larger, partly-unrelated diff; arguably more correct/current).
- C) **Full regenerate from the spreadsheet revision the committed files were
  last generated from** — removes `defaults` only, no content drift, but requires
  locating that exact spreadsheet revision.

**Decision**: **A — Surgical strip.** WM-27's scope is the defaults mechanism;

### RESOLVED: Should config resolution be de-duplicated into a shared helper?
**Context**: The `Object.assign(getDefaultConfig(), preset, getUrlConfig())`
merge currently lives in `simulation.ts`; `stores.ts` separately duplicates the
preset-name resolution `getUrlConfig().preset || getDefaultConfig().preset` it
feeds to `SimulationModel`. The bridge now needs the same merge (and the same
preset-name resolution). We can add a shared `getResolvedConfig()` to `config.ts`
and have all three call it, or resolve config bridge-locally and leave the
existing duplication untouched. The former touches files outside the Hazbot
directory; the latter keeps the change contained but adds a third copy of the
resolution logic.
**Options considered**:
- A) Add `getResolvedConfig()` to `config.ts`; refactor `simulation.ts`,
  `stores.ts`, and the bridge to use it.
- B) Resolve config inside the bridge only; leave `simulation.ts` / `stores.ts`
  as-is.
- C) Add the helper to `config.ts` and use it in the bridge, but leave
  `simulation.ts` / `stores.ts` for a separate cleanup.

**Decision**: **A — shared `getResolvedConfig()` in `config.ts`**, with an
enhancement so it serves all three call sites:

- `getResolvedConfig()` takes an **optional options object** — no raw boolean
  parameters.
- Default (no options / `presetOnly` unset): returns the full merge
  `Object.assign(getDefaultConfig(), presets[preset], getUrlConfig())` — used by
  the wildfire bridge and `simulation.ts`.
- With the options object's optional `presetOnly` key set: returns just the
  selected preset's partial config (`presets[urlConfig.preset || defaultPreset]`)
  — used by `stores.ts`, which passes the preset partial into `SimulationModel`.

The preset name is resolved the same way everywhere: `getUrlConfig().preset ||
getDefaultConfig().preset`.

**Superseded in part** (see the Self-Review *Senior Engineer* "Req 1 call-site"
finding): the `presetOnly` variant above is dropped. It was designed to return
the preset partial for `stores.ts` to pass into the `SimulationModel`
constructor — but that constructor hand-off is itself removed (`stores.ts` now
constructs `SimulationModel` with no argument, and the helper resolves the preset
from the URL). `getResolvedConfig()` therefore needs no options object; it takes
only an optional explicit preset partial for test injection. The core decision —
option A, a shared helper in `config.ts` — stands.

### RESOLVED: Is per-activity preset/URL verification in scope?
**Context**: The spreadsheet verification above confirms a matching preset

**Options considered**:
- A) Out of scope — assume LARA bindings are correct; this story only changes the
  *source* of defaults, not the activity setup.
- B) In scope — add a documented mapping of activity page → preset → URL params
  and verify each against SIMINIT as part of this story.
- C) Out of scope here, but file a follow-up ticket to audit LARA activity URLs.

**Decision**: **A — Out of scope.** Selecting the correct preset and URL params

#### RESOLVED: Requirement 1's three "call sites" are not independent — routing `simulation.ts` through `getResolvedConfig()` makes the `stores.ts` preset hand-off dead
Requirement 1 and RESOLVED OQ3 describe `simulation.ts`, `stores.ts`, and the
wildfire bridge as three call sites that "all route through" `getResolvedConfig()`
— `simulation.ts` and the bridge taking the full merge, `stores.ts` taking the
`presetOnly` partial. But `simulation.ts` and `stores.ts` are not independent
peers; they are the two ends of one dataflow. `stores.ts` is the only constructor
of `SimulationModel` (`new SimulationModel(presets[getUrlConfig().preset ||
getDefaultConfig().preset])`, `src/models/stores.ts:29`); the constructor's only
use of that argument is to thread it to `load(presetConfig)`
(`src/models/simulation.ts:68`), and `load()` — its sole call site — performs the
full merge `Object.assign(getDefaultConfig(), presetConfig, getUrlConfig())`
(`src/models/simulation.ts:162`).

So the two requirement clauses collide:
- If `load()` becomes `this.config = getResolvedConfig()` (the full-merge form,
  which re-resolves the preset from `getUrlConfig().preset` itself), the
  `presetConfig` argument threaded `stores.ts` → constructor → `load()` is never
  read. `getResolvedConfig({ presetOnly: true })` then has **no consumer**, and
  OQ3's entire `presetOnly` variant is dead on arrival.
- If `load()` keeps consuming its `presetConfig` argument, `simulation.ts` is
  *not* "routing the full merge through the helper" — it still does its own
  `Object.assign`, and only the bridge uses the full-merge form.

requirement should decide the fate of the `SimulationModel` constructor/`load()`
`presetConfig` parameter — drop it (then `stores.ts` has nothing to route and the
`presetOnly` variant exists only to feed a parameter that no longer exists), or
keep it (then `simulation.ts` does not use the helper for the merge). Note
`load()` is currently called exactly once, from the constructor, so no runtime
preset-switching use forces the parameter to stay.

**Resolution:** Requirement 1 was rewritten. `getResolvedConfig()` is a
full-merge helper that resolves the preset from the URL by default and accepts
an optional explicit preset partial for callers that inject a config (the
`SimulationModel` constructions in `simulation.test.ts`). `simulation.ts`'s
`load()` and the wildfire bridge route through it; `stores.ts` deletes its
duplicated `getUrlConfig().preset || getDefaultConfig().preset` expression and
constructs `SimulationModel` with no argument. The `presetConfig` parameter
becomes optional, retained only for test injection. This eliminates OQ3's
`presetOnly` variant — it existed solely to feed the `stores.ts` →
`SimulationModel` constructor hand-off, which is now removed rather than kept.
OQ3's *Decision* and the earlier "duplicated merge" Senior Engineer item each
gained a superseding note; OQ3's core decision (shared helper in `config.ts`)
stands.

terrain/veg/drought + wind) the classifier compares every `set*` variable
against. Surfacing the derived defaults themselves (in the sidebar, and/or the
activation payload) is also cheap — the bridge already computes them — and
covers the URL-only and override cases the preset name misses. Consider
broadening Requirement 13 to surface the derived `WildfireDefaults`, or record
explicitly why the preset name alone is accepted as sufficient for now (e.g. the
Hazbot is dev-only with no students yet — the same rationale already used to
down-rate this concern's urgency).

**Resolution:** Requirement 13 stays preset-name-only — broadening it to render
the full derived `WildfireDefaults` would grow a requirement that is itself
already added scope (see the *Product Manager* item above). A closing paragraph
was added to Requirement 13 recording the limitation explicitly: the preset name
does not catch a wrong baseline derived from URL-param overrides or a
`preset`-less activity, this is deliberately deferred because the Hazbot is
dev-only with no students yet, and surfacing the full derived defaults is a
natural candidate to revisit when the Hazbot becomes student-facing.

A later external review raised a lighter-weight variant — logging a stable
hash/fingerprint of the derived `WildfireDefaults` in the
`AnalysisEngineActivated` payload (cheaper than the full defaults, since the
bridge already computes them) to close the URL-override / `preset`-less blind
spot for log analysis. It was declined on the same grounds: it expands a
requirement that is itself already added scope, a bare hash is not
human-readable without a separate lookup of what differs, and the
dev-only/no-students rationale that defers full-defaults surfacing applies
equally. It belongs with the same student-facing follow-up.

### WCAG Accessibility Expert

#### RESOLVED: The match/no-match styling Requirement 13 reuses has no screen-reader equivalent
Requirement 13's *Dev sidebar* sub-bullet styles the resolved preset with the
sidebar's existing match/no-match convention and asserts: "That convention is
already double-encoded (colour + strikethrough) for colour-blind accessibility,
so reusing it needs no extra cue." The convention is `hazbot-sidebar-leaf-true`
(green + `text-decoration: underline`) vs `hazbot-sidebar-leaf-false`
(red + `text-decoration: line-through`)
([sidebar.css](../../src/hazbot/engine/sidebar/sidebar.css),
[expression-renderer.tsx:18](../../src/hazbot/engine/sidebar/expression-renderer.tsx#L18)).
For a *colour-blind* user the claim holds — underline vs line-through is a
non-colour visual cue. For a *screen-reader* user it does not: `text-decoration`
(underline or line-through) is not announced by screen readers, and the sidebar
applies these leaf classes with no `aria-label`, `aria-hidden`, visually-hidden
text, or paired ✓/✗ glyph (the ✓/✗/· glyphs exist only on category rows,
[sidebar.tsx:148](../../src/hazbot/engine/sidebar/sidebar.tsx#L148), not on
leaves). So a screen-reader user hears the preset name announced identically
whether it resolved to a known preset or silently fell back to the base config —
the exact mis-binding Requirement 13 exists to surface. Scope the "needs no extra
cue" claim to colour-blindness, and either add a text equivalent for the
no-match state (a visually-hidden label, an `aria-label`, or an appended
"(unrecognized preset)" string) or explicitly accept the screen-reader gap as
tolerable for a dev/researcher-only tool and say so.

**Resolution:** Requirement 13's *Dev sidebar* sub-bullet was reworded: the
double-encoding claim is now scoped to *colour-blind* accessibility, and the
no-match state must additionally carry a **text** cue — the displayed name
appends a short literal label (e.g. ` (unrecognized preset)`) so the base-config
fallback is announced by screen readers and visible regardless of styling. This
fixes only the one new element WM-27 adds; retrofitting the sidebar's broader
leaf-styling accessibility is out of scope.

#### RESOLVED: Requirement 13's sidebar preset display specifies a value and its styling but no identifying label
Requirement 13's *Dev sidebar* sub-bullet fully specifies the displayed *value*
(the URL `preset` string), its match/no-match colour-plus-decoration styling,
and — after the prior WCAG item — a text cue (` (unrecognized preset)`) for the
no-match state. What it never specifies is a **label** identifying what the value
is. As written, an implementer could satisfy Requirement 13 by rendering a bare
token — `plainsTwoZone`, or `badname (unrecognized preset)` — with nothing
stating it is the *resolved preset*.

For a screen-reader user this reproduces, in miniature, the gap the prior WCAG
item closed: the announced value has no context — the reader hears a token with
no field name. For a sighted developer the token is likewise ambiguous against
the sidebar's other rows. The prior item ensured the *no-match state* is conveyed
without styling; this item asks that the *value's identity* is conveyed at all.
Requirement 13 should state the sidebar display carries a short identifying label
(e.g. "Resolved preset:") preceding the value, so both the field and its
match/no-match state are intelligible without relying on position or styling.

**Resolution:** Requirement 13's *Dev sidebar* sub-bullet now states the display
carries a short identifying label (e.g. "Resolved preset:") preceding the value,
following the sidebar's existing label convention, so the field and — with the
text cue — its match/no-match state are intelligible without relying on position
or styling. The exact label wording and markup are left to implementation.
